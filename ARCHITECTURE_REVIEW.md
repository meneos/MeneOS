# MeneOS 架构审查报告（仅架构视角）

日期：2026-03-15  
范围：仅讨论架构设计与可演进性，不讨论安全性

## 结论摘要

当前系统已经具备微内核原型的关键形态：用户态服务化、IPC 通信、基础启动编排、用户态驱动尝试。  
但从“长期可演进平台”视角看，架构层面仍存在以下主要短板：

1. 分层职责边界未完全收敛
2. 进程生命周期控制面分散
3. 启动与服务发现硬编码较重
4. IPC 协议形态碎片化
5. 内存与 VMM 职责重叠
6. 设备模型链路未闭环
7. 运行时监督与恢复模型不足
8. ABI 演进边界尚未统一
9. 多架构声明与核心机制实现不对齐

---

## 1. 分层职责边界未完全收敛

### 现象
- mene-kernel 作为核心编排层存在，但 memory/trap 仍是占位或空实现。
- 真实功能散落在 mene-syscall、mene-memory、mene-task 中。

### 架构影响
- 新功能难以判断归属，长期会形成“按便利堆代码”的架构漂移。
- 模块间依赖关系容易变成环状，维护成本上升。

### 建议
- 先定义稳定分层：
  - Kernel Primitive 层：调度、地址空间原语、IPC 原语、中断陷入。
  - Kernel Facade 层：统一 syscall 入口与对象路由。
  - User Service 层：FS、VMM、Driver、Init 的策略逻辑。
- 给每层写“只允许依赖谁”的硬规则，配合 CI 检查。

---

## 2. 进程生命周期控制面分散

### 现象
- ProcessInfo 与 PID 在 process 模块。
- 创建流程在 syscall 与 task 装载路径。
- 退出清理在 syscall 分支中完成。

### 架构影响
- 生命周期事件（创建、崩溃、退出、重启、统计）没有统一所有者。
- 后续做调度策略、可观测性、服务编排时，控制逻辑会越来越分叉。

### 建议
- 引入 Process Supervisor（内核内或特权服务）作为统一生命周期控制面。
- 进程状态机规范化：Created -> Ready -> Running -> Exited/Faulted。
- 统一事件总线：spawn、exit、fault、restart 进入单一事件流。

---

## 3. 启动编排是脚本化硬编码，而非服务图模型

### 现象
- 启动顺序依赖 boot.toml 的服务图配置。
- 服务角色由路径名称隐式表达。

### 架构影响
- 难以支持多启动 Profile（开发/生产/测试）。
- 难以表达依赖拓扑、并行启动、失败重试与降级策略。

### 建议
- 将 boot.toml 演进为“服务清单 + 依赖图 + 启动策略”：
  - name
  - binary
  - depends_on
  - restart_policy
  - health_probe
- Init 从“顺序脚本执行器”升级为“拓扑调度器”。

---

## 4. 服务发现与系统句柄硬编码较重

### 现象
- 固定句柄号与固定路径映射并存。
- 能力注入逻辑对服务路径有显式耦合。

### 架构影响
- 动态服务、多实例服务、版本化服务难以自然扩展。
- 服务重启后句柄语义与发现语义不易保持一致。

### 建议
- 引入 Service Registry 抽象：
  - 注册：service_name -> endpoint/capability
  - 查询：按名称、版本、实例标签
  - 订阅：服务上下线事件
- 把启动时的静态注入迁移到“注册后分发”。

---

## 5. IPC 协议层次偏低，业务层重复定义报文

### 现象
- 内核 IPC 仅提供基础消息与可选能力传递。
- FS、BLK、VMM 在用户态分别实现自己的请求头、tag、重试语义。

### 架构影响
- 协议碎片化，跨服务复用和统一调试成本高。
- 客户端库随服务数量增长而线性膨胀。

### 建议
- 在 ulib 之上定义统一 IPC 框架层：
  - 标准头：op, req_id, flags, status
  - 统一错误码映射
  - 统一 timeout/cancel 语义
  - 可选的零拷贝缓冲协商接口
- 业务服务仅定义 payload，不再重复定义通用机制。

---

## 6. 内存子系统与 VMM 服务职责重叠

### 现象
- 内核 syscall 提供 mmap/device map 等直接能力。
- 同时存在 VMM 服务和专用映射调用。

### 架构影响
- 不清楚“谁负责策略，谁负责原语”，导致系统行为分散。
- 未来要支持复杂地址空间策略时，会出现双轨实现。

### 建议
- 明确单一原则：
  - 内核只提供最小原语（map/unmap/query）。
  - VMM 服务统一承担策略（布局、回收、配额、统计）。
- 把策略型接口逐步收敛到 VMM 服务，内核接口保持稳定小集合。

---

## 7. 设备模型发现链路与驱动绑定链路未闭环

### 现象
- 内核已有设备树解析与驱动注册骨架。
- 用户态驱动仍包含平台固定地址常量路径。

### 架构影响
- 新平台移植时，驱动代码需要改动而不是仅替换设备描述。
- “发现 -> 分配 -> 绑定 -> 运行时状态”缺少统一数据通道。

### 建议
- 设备资源描述对象化：
  - device_id
  - resource list（mmio、irq、dma）
  - bind state
- 由内核/设备管理服务发布资源描述，驱动按描述绑定，不直接硬编码平台常量。

---

## 8. 运行时监督与恢复模型不足

### 现象
- 进程退出后主要是表项删除，缺少系统级监督者语义。

### 架构影响
- 核心服务故障后恢复机制依赖人工或上层偶发逻辑。
- 无法构建可预测的服务可用性目标。

### 建议
- 引入 Supervisor 策略：
  - one-shot
  - always
  - on-failure
- 增加依赖联动恢复策略与冷却时间窗口。
- 为每个服务定义健康探针与重启上限。

---

## 9. ABI 演进边界尚未统一

### 现象
- 同一 syscall 入口同时处理 Linux 风格调用与 Mene 自定义调用。

### 架构影响
- 兼容层与原生层耦合，后续维护和行为收敛困难。

### 建议
- 拆分语义层：
  - Native ABI（Mene）
  - Compatibility ABI（POSIX/Linux 子集）
- 内部路由统一为对象操作，不在入口层混杂策略。

---

## 10. 多架构支持声明完整，但关键机制一致性不足

### 现象
- 配置层有多架构分支。
- 核心 trap/memory 控制路径仍未形成稳定统一实现面。

### 架构影响
- 易出现“配置上支持，行为上不一致”的维护债务。

### 建议
- 定义跨架构最小一致语义：
  - trap event model
  - syscall return contract
  - page fault handling contract
- 每个架构实现都通过同一组内核一致性测试。

---

## 目标状态建议（不涉及安全项）

建议将目标架构抽象为三层：

1. Microkernel Core
- 线程调度
- IPC 原语
- 地址空间原语
- trap/interrupt 统一分发

2. System Control Plane
- Process Supervisor
- Service Registry
- Device Manager
- VMM Policy Service

3. User Services
- FS
- Serial
- Virtio Block
- Init/Session Manager

这种分法的核心价值是：
- Core 保持小且稳定
- 策略集中在控制面
- 业务服务可替换、可扩展、可回归测试

### 与现有代码模块的落地映射（建议）

1) Microkernel Core（只保留原语）
- 对应模块：`kernel/mene-kernel`、`kernel/mene-task`、`kernel/mene-ipc`、`kernel/mene-memory`、`kernel/mene-trap`、`kernel/mene-syscall`
- 约束：
  - 仅暴露“线程/地址空间/IPC/trap”最小原语。
  - 不承载服务策略（如重启策略、服务发现策略、设备分配策略）。
  - syscall 层只做能力检查与对象路由，不做业务编排。

2) System Control Plane（集中策略与编排）
- 对应模块：优先作为特权系统服务落在 `apps/init`（后续可拆分 supervisor/registry/device-manager/vmm-policy 四个服务）。
- 约束：
  - 统一管理进程生命周期、服务注册发现、设备资源分配、VMM 策略。
  - 通过 IPC 调用 Core 原语，不新增“旁路内核接口”。
  - 失败恢复、依赖拓扑、版本选择等都在控制面收敛。

3) User Services（业务能力服务化）
- 对应模块：`apps/fs`、`apps/serial`、`apps/virtio_blk`、`apps/init`
- 约束：
  - 服务只关注领域逻辑（文件、串口、块设备、会话管理）。
  - 通过 Service Registry 获取依赖，不硬编码固定句柄与固定路径。
  - 升级/替换通过协议兼容与注册信息完成，不要求改内核。

### 分层边界判定规则（可用于代码审查）

- 若代码回答的是“如何调度/映射/分发中断”，归 Core。
- 若代码回答的是“谁启动谁、失败如何恢复、资源如何分配”，归 Control Plane。
- 若代码回答的是“如何实现具体业务能力”，归 User Services。
- 若一个改动同时触及两层及以上，默认先拆分为“原语接口变更 + 控制面策略变更 + 服务实现变更”三个提交。

---

## 三阶段演进路线

### 里程碑 1：边界收敛（1-2 周）
- 固化模块职责文档
- 收敛 process 生命周期入口
- 明确内核原语接口最小集合

### 里程碑 2：控制面成型（2-4 周）
- 落地 Service Registry
- 落地 Supervisor 与服务状态机
- 将启动脚本演进为依赖图调度

### 里程碑 3：协议与驱动统一（2-4 周）
- 统一 IPC 框架头与错误语义
- 驱动资源描述对象化
- 完成跨架构一致性测试基线

---

## 最终评价

MeneOS 当前阶段是一个方向正确、工程上已经可运行的微内核原型。  
接下来最关键的不是继续堆功能，而是先完成“架构控制面”与“分层边界”的收敛。  
一旦这两件事完成，后续功能扩展（文件系统增强、网络栈、更多驱动、多架构稳定化）都会顺畅很多。
